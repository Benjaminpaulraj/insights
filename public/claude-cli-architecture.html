How Claude CLI
Agent Works

A visual, plain-English walkthrough of every layer — from the moment you press Enter to the moment output appears on your screen.

TL;DR — 6 Key Facts

  • Claude CLI is a 10-layer orchestration engine, not just a chatbot wrapper
  • Every prompt flows through: pre-processing → permission checks → context assembly → model API → tool execution
  • Tools are Claude's "hands" — 40+ tools across 8 categories (File, Shell, Search, Web, Agent, MCP, Task, Scheduling)
  • Sub-agents are worker processes with their own 200K token context windows — isolated from yours
  • Memory lives in: CLAUDE.md (permanent), ~/.claude/memory/ (AI-maintained), git history (implicit)
  • Skills are pre-built Markdown prompt templates — not code — loaded on demand when you type /skill-name

The Complete Flow

Request Lifecycle

Every prompt passes through all 10 layers below. Scroll down to watch each layer animate in, or click any card to explore it in depth in the Component Explorer.

⌨️ Layer 1

User Input

Your message enters via one of 5 modes: plain text · ! shell bypass · /skill template · @file attachment · STDIN pipe

📋 Layer 2

Pre-Processing

CLAUDE.md is loaded. Memory files are retrieved. Git status is injected. System context (date, email) is assembled — all before any API call.

🔒 Layer 3

Permission System

Every tool call is checked against allowlists in settings.json and settings.local.json. Hooks (PreToolUse, PostToolUse) fire here.

📦 Layer 4

Context Assembly

System prompt + tool schemas + conversation history + your message are combined into one payload. Deferred tool schemas are loaded on-demand via ToolSearch.

🧠 Layer 5

Claude Model API

Context sent to Anthropic API. Response streams back token-by-token. Model tier: Haiku 4.5 / Sonnet 4.6 (default) / Opus 4.7. Plan mode and Fast mode alter this layer.

🔀 Layer 6

Response Parsing

Plain text → rendered as Markdown. Tool call blocks → routed to executor. Multiple independent calls → run in parallel. AskUserQuestion → interactive UI prompt.

🔧 Layer 7

Tool Execution

The agent's "hands." 40+ tools across 8 categories: File · Shell · Search · Web · Agent · MCP · Task · Scheduling. Results injected back into context for next turn.

🤖 Layer 8

Sub-Agent System

Parent agents spawn child agents with their own 200K context window. Two isolation modes: worktree (git-isolated) or shared (same directory). Children run independently.

💾 Layer 9

Memory System

Three-tier persistence: CLAUDE.md (permanent project rules) + Memory files (~/.claude/projects/.../memory/) + Git history (implicit queryable timeline).

📺 Layer 10

Output Layer

Markdown rendered in terminal. File changes applied to disk. Tool results shown. Background tasks monitored. Loop continues until no more tool calls are needed.

Layer 1 — User Input

The harness detects which input mode you're using before passing anything to Claude. This routing happens instantly, in the CLI process — no model call needed.

5 Input Modes

  • Plain text — full agent pipeline, most common
  • ! prefix — direct shell bypass, no model call
  • /skill-name — loads a skill template first
  • @file — attaches file content to the message
  • STDIN pipe — piped text injected as context

Key Facts

  • The ! prefix is free — zero API cost
  • STDIN output appears in the conversation
  • Multiple @file attachments are supported
  • Skill invocation adds template to context before processing
  • Input is never sent to the model raw — always assembled first

Layer 2 — Pre-Processing

Four data sources are assembled into the system prompt before any API call is made. This is free — no tokens are billed until the assembled payload hits the model.

Sources Assembled

  • CLAUDE.md — project rules, always loaded from root
  • MEMORY.md — index loaded; relevant memory files retrieved
  • Git status — branch, staged files, recent commits
  • System context — current date, user email, model ID

Files Involved

  • ./CLAUDE.md
  • ~/.claude/CLAUDE.md
  • ~/.claude/projects/…/MEMORY.md
  • ~/.claude/projects/…/memory/*.md

Layer 3 — Permission System

No tool call reaches the executor without passing through the permission system. Allowlists are checked, hooks are fired, and the user is prompted if needed.

Settings Hierarchy

  • ~/.claude/settings.json — user-level (global)
  • .claude/settings.local.json — project-level (overrides global)
  • Pattern matching: Bash(git *), Read, Edit
  • Deny list also supported for explicit blocking

4 Hook Types

  • PreToolUse — fires before every tool call
  • PostToolUse — fires after every tool result
  • Stop — fires when Claude finishes responding
  • Notification — fires on system alerts

Layer 5 — Claude Model API

The assembled context is sent to Anthropic's API. Three model tiers. Two special modes. Response streams token-by-token — you see words appear as they're generated.

Model Tiers

  • Haiku 4.5 — fastest, great for sub-agents and simple tasks
  • Sonnet 4.6 — balanced, the default for most tasks
  • Opus 4.7 — most capable, best for hard reasoning

Special Modes

  • /plan — Plan Mode: read → design → await approval before any edits
  • /fast — Fast Mode: Opus with faster output (not a smaller model)
  • Streaming enabled by default — output appears word-by-word
  • Multi-turn: each tool call starts a new API turn

Layer 7 — Tool Execution

40+ tools across 8 categories. Tool results are injected back into the context as the next "turn" — the model sees what happened and decides what to do next.

Tool Categories

  • 📁 File — Read, Write, Edit
  • 🖥️ Shell — Bash (foreground + background)
  • 🔍 Search — Grep, Glob, LS, Find
  • 🌐 Web — WebFetch, WebSearch
  • 🤖 Agent — spawn sub-agents, SendMessage
  • 🔌 MCP — Chrome DevTools, Google Drive, custom
  • 📋 Task — Create, Get, List, Stop, Update
  • ⏰ Schedule — CronCreate, ScheduleWakeup

Key Behaviours

  • Independent calls execute in parallel — free speed
  • Each call result is a new "turn" for the model
  • Background Bash persists across multiple turns
  • Deferred MCP/scheduling tools need ToolSearch first
  • The Bash tool is most powerful — and most carefully permissioned

Layer 9 — Memory System

Claude has no built-in persistence. Three external systems simulate long-term memory — each with different scope, lifetime, and cost.

Tier 1 — CLAUDE.md

  • Human-written, version-controlled
  • Always loaded at session start
  • Counts against your 200K budget every turn
  • Best for project rules and conventions

Tier 2 — Memory Files

  • AI-written and maintained across sessions
  • Stored in ~/.claude/projects/.../memory/
  • MEMORY.md is the index (always loaded)
  • Best for decisions, preferences, project state

Tier 3 — Git History (Implicit)

  • Every commit, every diff — queryable via Bash
  • No configuration needed
  • Unlimited depth — access any point in project history

Layer 10 — Output Layer

Output is not just text. Responses route to different destinations depending on their type. The loop continues until no more tool calls remain.

Output Destinations

  • Plain text → terminal as GitHub Markdown
  • File edits → applied directly to disk
  • Tool results → injected into next context turn
  • Permission prompts → interactive terminal UI
  • Background tasks → task queue (TaskGet to monitor)
  • Push notifications → system desktop alerts

The Loop

  • After each tool call, results return to the model
  • Model decides: done → output final text, or continue → more tool calls
  • Loop repeats until no tool calls remain
  • Context compaction fires if window approaches limit
  • ScheduleWakeup can restart the loop after a delay

How Spawning Works

Worktree Mode

  • Agent gets its own auto-created git branch
  • File changes are isolated — parent can't see them until merged
  • Auto-cleaned if the agent makes no changes
  • Safe for experimental or parallel work

Shared Mode (default)

  • Agent works in the same directory as parent
  • File changes are immediately visible to all agents
  • Good for read-only searching and exploration
  • Use with care when multiple agents write files

Layer 9

Memory System

Claude has no built-in persistence. Three external tiers simulate long-term memory — each with different scope, lifetime, and cost.

Tier 1 — Permanent

CLAUDE.md

Human-written project instructions. Loaded automatically at every session start. Version-controlled alongside your code.

⚠️ Counts against your 200K token budget on every turn. Keep it focused.

# CLAUDE.md ## Project Astro 6 blog deployed at /insights ## Commands npm run dev # localhost:4321 npm run build # → /dist ## Rules - No jargon without explanation - BASE_URL prefix all links
Tier 2 — AI-Maintained

Memory Files

Written and updated by Claude across sessions. MEMORY.md is the index — always loaded. Individual files retrieved on demand.

Best for: decisions, preferences, project state, "remember that we chose X."

~/.claude/projects/blog/memory/ ├── MEMORY.md ← index ├── decisions.md ← tech choices ├── architecture.md ← system design └── user-prefs.md ← how you work # MEMORY.md (excerpt) - [Decisions](decisions.md) Postgres over SQLite for scale - [Prefs](user-prefs.md) User prefers async/await
Tier 3 — Implicit

Git History

Every commit, every diff — fully queryable via the Bash tool. No configuration needed. Unlimited depth and detail.

The most powerful memory system: it's always there, it's always accurate, and it never costs context budget until you query it.

Model Context Protocol

MCP Server Connections

MCP lets external servers plug into Claude's tool layer. Hover any node to highlight the connection.

How MCP Works

Connection Model

  • MCP servers run as separate processes on your machine
  • Claude discovers them via config in settings.json
  • Their tool schemas are exposed as deferred tools (loaded via ToolSearch)
  • Communication is bidirectional over stdio or HTTP

Available MCP Servers

  • Chrome DevTools — debug, screenshot, audit, automate
  • Google Drive — read and write Google Docs, Sheets
  • Custom servers — any MCP-compatible server you build
  • Community registry growing at modelcontextprotocol.io
